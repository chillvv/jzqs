# 售后台账与结算中心设计

**需求目标**

- 将后台售后拆成两个使用视角：
  - `售后登记`：负责记录问题和补充信息
  - `售后处理`：负责人工判断、结算执行和结果回查
- 商家可以按指定日期查看全部订单，并对选中的订单手动登记售后。
- 列表要能清楚看到：`原因`、`参数`、`用户`、`处理结果`，方便商家快速判断。
- 秒退款类售后需要保留，但默认弱化展示，避免干扰人工真正要处理的售后。
- 顾客餐包变动和损失餐数统计都在`结算确认`时才正式生效，登记阶段只做记录。
- 第一版补齐两类基础补偿能力：
  - `补零餐`
  - `果蔬汁`

**现状**

- 后台现有 [AftersalePage.tsx](file:///d:/Code/jzqs/admin/src/modules/aftersales/AftersalePage.tsx) 只有统一列表和处理弹窗，没有独立的登记入口。
- 后端现有 [AftersaleServiceImpl.java](file:///d:/Code/jzqs/backend/src/main/java/com/jzqs/app/aftersale/service/impl/AftersaleServiceImpl.java) 已支持：
  - 后台直接创建售后
  - 用户发起售后
  - 后台处理售后
- 当前模型以 `walletDelta` 为主，没有独立的`预估损失餐数`和`结算损失餐数`表达。
- 当前列表筛选偏单一，只支持单日 `serveDate`，不适合区分“秒退款杂项”和“人工重点售后”。

**核心设计**

- 后台只保留一个售后主模块，但拆成两个页签或两个子页面：
  - `售后登记`
  - `售后处理`
- 数据层优先复用现有 `aftersale_cases` 主表，不新增第二套主表。
- “台账”和“结算中心”的区别主要体现在：
  - 页面视图不同
  - 状态字段不同
  - 允许的动作不同
- 第一版不做复杂审批流，不引入多级审核；采用“登记 -> 待处理 -> 已结算/已驳回”的轻量状态模型。
- 顾客餐包、钱包流水、损失统计只在`结算`阶段执行，避免登记错误直接影响资产。

**页面结构**

### 售后登记

- 主要用途：
  - 商家手动指定日期订单登记售后
  - 查看用户主动申请后流入后台的待处理台账
- 页面结构建议：
  - 顶部操作区：`登记售后` 按钮
  - 台账列表区：展示近期台账
  - 右侧或弹窗：登记表单
- `登记售后` 流程：
  - 先选 `serveDate`
  - 拉取该日全部订单
  - 选中订单
  - 填写售后类型、原因、参数、预估损失餐数、备注
  - 提交后生成台账记录
- 登记页不做复杂筛选，只保留：
  - `日期范围`
  - `状态`
  - `隐藏秒退款`
- 登记页列表重点展示字段：
  - 用户姓名
  - 手机号
  - 用餐日期
  - 餐次
  - 订单状态
  - 售后来源
  - 原因
  - 参数摘要
  - 预估损失餐数
  - 当前状态

### 售后处理

- 主要用途：
  - 聚焦人工真正需要判断和执行的售后
  - 查看已处理记录和结算结果
- 页面结构建议：
  - 顶部筛选区
  - 待结算/已结算列表
  - 结算抽屉或弹窗
- 默认列表策略：
  - 默认开启 `隐藏秒退款`
  - 默认优先展示 `待结算`
  - `秒退款自动完成` 记录通过开关显式查看
- 处理页重点展示字段：
  - 用户
  - 售后原因
  - 问题参数
  - 预估损失餐数
  - 最终结算结果
  - 餐包变动数量
  - 补偿项
  - 结算备注
  - 结算时间
  - 处理人

**筛选策略**

- 结合当前业务单量较少，第一版只保留高价值筛选，不做重搜索系统。
- 两个页面统一保留以下筛选能力：
  - `开始日期`
  - `结束日期`
  - `状态`
  - `售后类型`
  - `隐藏秒退款`
- `隐藏秒退款` 默认开启。
- 关键词搜索保留简单模式，只搜索：
  - 用户名
  - 手机号
  - 原因文本
- 不额外增加复杂组合筛选、分组、导出等能力。

**秒退款策略**

- 秒退款类售后继续保留在售后体系内，不单独拆新模块。
- 对于“前一天预定、前一天取消、系统自动秒退款”的记录，新增独立来源或标签：
  - `AUTO_REFUND`
- 这类记录默认不进入人工处理主视图的优先列表。
- 页面上通过：
  - `隐藏秒退款` 开关
  - 来源标签
  - 默认折叠或弱化样式
  来降低干扰。
- 秒退款仍应保留详情和回查能力，避免账务对不上时无迹可查。

**状态模型**

- 建议统一使用以下状态：
  - `LOGGED`：已登记，尚未进入人工处理
  - `PENDING_SETTLEMENT`：待结算
  - `SETTLED`：已结算，结果已生效
  - `REJECTED`：已驳回，不执行补偿
  - `AUTO_REFUNDED`：系统秒退款自动完成
- 用户主动申请售后和商家手动登记售后都先进入：
  - `LOGGED` 或 `PENDING_SETTLEMENT`
- 如果系统判断属于秒退款路径，则直接进入：
  - `AUTO_REFUNDED`
- 第一版不保留 `PROCESSING` 这类中间态，除非实际实现时现有代码迁移成本过高。
- 如果为了兼容现有代码短期保留旧状态值，前端展示层需要映射成以上业务状态文案。

**数据模型**

- 第一版优先复用 `aftersale_cases`，新增字段表达“登记信息”和“结算信息”。
- 建议新增或补齐以下字段：
  - `ledger_status`
  - `settlement_status`
  - `source_category`
  - `issue_param_summary`
  - `estimated_loss_meals`
  - `settled_loss_meals`
  - `settlement_action`
  - `meal_wallet_delta`
  - `gift_zero_meal_count`
  - `gift_veggie_juice_count`
  - `settlement_remark`
  - `settled_at`
  - `settled_by`
- 字段语义：
  - `issue_param_summary`：给商家看的简明参数描述，比如“晚到 35 分钟”“餐品破损 1 份”
  - `estimated_loss_meals`：登记时预估损失
  - `settled_loss_meals`：结算确认后最终损失
  - `meal_wallet_delta`：实际增减餐包数量
  - `gift_zero_meal_count`、`gift_veggie_juice_count`：非钱包型补偿项数量
- 如果数据库当前不方便拆 `ledger_status` 和 `settlement_status` 两个字段，第一版可先只保留一个主状态字段，并在服务层控制行为，但出参仍要按“登记态”和“结算态”输出。

**动作模型**

- 结算中心支持以下结算动作：
  - `REFUND_TO_WALLET`：退回顾客餐包
  - `COMPENSATE_MEALS`：补回顾客餐包
  - `REGISTER_ONLY`：仅登记，不做资产补偿
  - `REJECT`：驳回
- 第一版允许“餐包补偿”和“赠品补偿”同时存在，但实现方式建议保持简单：
  - 主结算动作保留一个
  - 赠品补偿项作为附加字段记录
- 示例：
  - 退回餐包 `1` 份，同时赠送果蔬汁 `1` 瓶
  - 不改餐包，只记录补零餐 `1` 份
- `补零餐` 在第一版仅作为运营补偿记录项，不影响顾客餐包。
- `果蔬汁` 在第一版仅作为补偿记录项，不做库存扣减联动。

**参数展示规则**

- 每条售后在列表和详情都需要有一段统一的参数摘要。
- 参数摘要示例：
  - `配送慢 35 分钟`
  - `餐品破损 1 份`
  - `漏送 1 份`
  - `退款退回餐包 1 份`
  - `果蔬汁 1 瓶`
- 第一版不做复杂结构化参数编辑器，后台表单仍以简单字段为主，最终拼出稳定摘要字符串即可。

**接口设计**

### 后台列表接口

- 扩展现有 `GET /api/admin/aftersales`
- 新增查询参数：
  - `startDate`
  - `endDate`
  - `hideAutoRefund`
  - `view`
- `view` 用于区分：
  - `ledger`
  - `settlement`
- 返回结果需补充：
  - 参数摘要
  - 预估损失餐数
  - 结算损失餐数
  - 餐包变动值
  - 补零餐数量
  - 果蔬汁数量
  - 来源分类
  - 结算动作

### 后台登记接口

- 继续使用或扩展现有 `POST /api/admin/aftersales`
- 新增入参：
  - `issueParamSummary`
  - `estimatedLossMeals`
  - `sourceCategory`
- 保持“创建只登记，不结算”的语义。
- 手动登记创建后默认进入：
  - `LOGGED` 或 `PENDING_SETTLEMENT`

### 订单选择接口

- 新增后台接口用于登记页拉取指定日期订单：
  - `GET /api/admin/aftersales/order-options?serveDate=YYYY-MM-DD`
- 返回信息建议包含：
  - 订单 id
  - 用户名
  - 手机号
  - 日期
  - 餐次
  - 当前订单状态
  - 地址摘要
- 第一版只支持单日拉单，不做更复杂的多天选单。

### 结算接口

- 扩展现有 `POST /api/admin/aftersales/{caseId}/resolve`
- 新增入参：
  - `settledLossMeals`
  - `giftZeroMealCount`
  - `giftVeggieJuiceCount`
  - `hideFromMainView`
- 结算接口负责：
  - 校验状态是否允许结算
  - 写入最终结算动作和备注
  - 在需要时调整顾客餐包
  - 写补偿项记录
  - 写结算时间和处理人

**结算执行规则**

- `登记` 阶段：
  - 不修改 `meal_wallets`
  - 不写最终损失统计
  - 不写结算类钱包流水
- `结算` 阶段：
  - 如果动作为 `REFUND_TO_WALLET` 或 `COMPENSATE_MEALS`，才修改顾客餐包
  - `settledLossMeals` 写入最终损失统计
  - `giftZeroMealCount` 和 `giftVeggieJuiceCount` 只记录补偿项
- `REGISTER_ONLY`：
  - 不影响顾客餐包
  - 允许写 `settledLossMeals`
- `REJECT`：
  - 不影响顾客餐包
  - 默认 `settledLossMeals = 0`
- 秒退款：
  - 保持原有自动退款能力
  - 但在展示层标记为 `AUTO_REFUNDED`

**前端体验**

### 登记页表单

- 建议包含字段：
  - 订单
  - 售后类型
  - 原因代码
  - 原因说明
  - 参数摘要
  - 预估损失餐数
  - 备注
- 表单校验：
  - 必选订单
  - 原因必填
  - 预估损失餐数不得为负
- 提交成功后：
  - 刷新列表
  - 给出成功提示
  - 自动回到新记录所在列表

### 处理页结算弹窗

- 建议包含字段：
  - 售后信息摘要
  - 结算动作
  - 餐包变动数量
  - 最终损失餐数
  - 补零餐数量
  - 果蔬汁数量
  - 处理备注
- 根据动作动态显示字段：
  - `REFUND_TO_WALLET` / `COMPENSATE_MEALS`：显示餐包变动数量
  - 任意动作都可额外填写补零餐数量和果蔬汁数量
  - `REGISTER_ONLY` / `REJECT`：只要求备注和最终损失餐数规则校验

**兼容策略**

- 现有用户侧售后申请入口保留，不改小程序主流程。
- 现有秒退款逻辑保留，不阻塞本次改造。
- 后台原有统一售后页可以先演进为：
  - 一个页面两个页签
  - 或两个路由页面复用同一套组件
- 为降低改造成本，第一版允许：
  - 数据库存量记录继续按旧字段展示
  - 新增字段仅对新记录完整生效
- 对旧数据缺失的新字段，前端按 `-` 或 `0` 回退展示。

**错误处理**

- 登记时如果订单已有未结案售后，继续阻止重复创建。
- 结算时如果记录已完成结算，接口需幂等返回，避免重复写餐包。
- 补偿项数量必须是非负整数。
- 如果顾客没有有效餐包账户：
  - `REFUND_TO_WALLET`
  - `COMPENSATE_MEALS`
  应返回可理解的业务错误。
- 补偿项写入失败不能默默吞掉，必须让商家知道结算未成功。

**测试策略**

- 后端测试重点：
  - 后台登记售后只写台账，不改餐包
  - 结算时才写餐包和最终损失餐数
  - 秒退款记录能按来源标签被过滤
  - 结算接口重复调用不重复写钱包流水
  - `补零餐` 和 `果蔬汁` 能正确保存并在列表返回
- 前端测试重点：
  - 登记页能按日期拉订单并提交台账
  - 处理页默认隐藏秒退款
  - 列表能展示原因、参数、用户、补偿结果
  - 结算动作切换时表单字段联动正确
  - 旧记录缺失新字段时页面不报错

**实施范围建议**

- 第一阶段：
  - 做“登记页 + 处理页”结构
  - 做日期范围和隐藏秒退款筛选
  - 做参数摘要与损失餐数字段
  - 做结算时生效的资产逻辑
- 第二阶段：
  - 完善赠品补偿展示
  - 视情况补充统计卡片或简单汇总
- 暂不纳入本期：
  - 复杂审批流
  - 库存联动
  - 导出报表
  - 多层级消息通知
