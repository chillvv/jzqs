后端逻辑调整方案

配送流程改为两步：先归区域 → 再分配骑手

当前问题

  • AREA_ONLY 模式实际上也自动选了骑手，不符合"只归区域"的需求
  • 待分配列表包含已分配订单（hasDispatchAssignment === true），需要过滤掉
  • 地址匹配用硬编码 keyword，区域关键词不可维护
  • 智能派单未考虑"只归区域"的场景

改动方案

1. 待分配列表 — 只返回真正未分配的订单

pendingItems() SQL 添加 AND da.id IS NULL，已存在 dispatch_assignments 的订单不再出现。

2. 批量分配改为"只归区域"

去掉 AREA_AND_RIDER 模式。AREA_ONLY 只创建 dispatch_assignments 记录（rider_name = NULL，status = 'AREA_ASSIGNED'），不分配骑手。
前端批量工具栏只保留"选择区域"→"归入区域"，去掉骑手选择。

3. 区域管理 — 分配骑手

新增逻辑：在区域管理中给区域分配骑手时，将该区域下所有 status = 'AREA_ASSIGNED' 的订单批量更新 rider_name 和 status = 'DISPATCHING'。
也可以单独给某订单分配骑手（弹窗选择）。

4. 地址匹配 — 关键词可维护

去掉 SQL 里的硬编码 UNION ALL keyword，改用 dispatch_area_bindings.keywords 字段进行 LIKE 匹配。
商家可以在区域管理中配置关键词，例如"高新区 → 科技路、软件园、创业大厦"。

数据流变化

待分配订单 → 归入区域 → AREA_ASSIGNED
AREA_ASSIGNED → 分配骑手（区域管理） → DISPATCHING
DISPATCHING → 送达回执 → DELIVERED

影响范围

后端修改
  • DispatchServiceImpl.java — pendingItems、batchAssign 逻辑
  • BatchAssignRequest.java — 简化，去掉 riderName
  • DispatchController.java — 新增/修改端点
前端修改
  • DispatchHomePage.tsx — 简化表格和工具栏
  • DispatchAreasPage.tsx — 融入配送看板和骑手分配
  • http.ts / types.ts — 调整 API 调用和类型
